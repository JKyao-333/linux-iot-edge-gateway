# 自定义串口协议说明

## 1. 协议用途

该协议用于 STM32 传感器终端与 Linux 边缘网关之间的串口通信。

协议支持：

- 多设备 ID
- 多种命令字
- 变长 Payload
- CRC16-Modbus 校验
- 半包与粘包解析
- 异常帧过滤和同步恢复

## 2. 帧格式

| 字段 | 长度 | 说明 |
|---|---:|---|
| 帧头 | 2 B | 固定为 `0xAA 0x55` |
| Payload 长度 | 1 B | Payload 的字节数，不包含其他字段 |
| 命令字 | 1 B | 表示消息类型 |
| 设备 ID | 1 B | 标识数据来源 |
| Payload | N B | 命令对应的数据内容 |
| CRC16 | 2 B | 低字节在前，高字节在后 |

完整帧长度为 `N + 7` 字节。

解析器允许的最大 Payload 长度为 64 字节。

## 3. 命令字

| 命令字 | 名称 | Payload 长度 |
|---|---|---:|
| `0x01` | 传感器周期数据 | 11 B |

其他命令字可以在后续版本中扩展，例如设备配置、时间同步和控制应答。

## 4. 传感器 Payload

所有多字节传感器字段使用大端字节序。

| 偏移 | 长度 | 字段 | 编码方式 |
|---:|---:|---|---|
| 0 | 2 B | 温度 | 有符号整数，实际值除以 10，单位摄氏度 |
| 2 | 2 B | 湿度 | 无符号整数，实际值除以 10，单位百分比 |
| 4 | 2 B | 气体浓度 | 无符号整数，单位 ppm |
| 6 | 2 B | 电池电压 | 无符号整数，单位 mV |
| 8 | 1 B | 设备状态 | 位标志 |
| 9 | 2 B | 数据序号 | 无符号整数，大端序 |

## 5. 状态字节

| 位 | 掩码 | 含义 |
|---:|---|---|
| bit0 | `0x01` | 传感器故障 |
| bit1 | `0x02` | 低电量 |
| bit2 | `0x04` | 设备正在校准 |
| bit3-bit7 | - | 保留 |

状态值为 `0x00` 表示设备状态正常。

## 6. CRC16-Modbus

CRC 参数：

- 初始值：`0xFFFF`
- 多项式：`0xA001`
- 计算范围：长度、命令字、设备 ID 和 Payload
- 不包含帧头
- 发送顺序：CRC 低字节在前，高字节在后

对于传感器数据帧，CRC 计算输入长度为 `1 + 1 + 1 + 11 = 14` 字节。

## 7. 正常帧示例

完整字节流：

`AA 55 0B 01 10 00 FD 02 60 01 2C 0E 74 00 00 01 4D 58`

字段解释：

| 字段 | 数据 | 解释 |
|---|---|---|
| 帧头 | `AA 55` | 固定帧头 |
| 长度 | `0B` | Payload 长度为 11 |
| 命令字 | `01` | 传感器周期数据 |
| 设备 ID | `10` | 十进制设备 ID 为 16 |
| 温度 | `00 FD` | 253，即 25.3 摄氏度 |
| 湿度 | `02 60` | 608，即 60.8% |
| 气体浓度 | `01 2C` | 300 ppm |
| 电池电压 | `0E 74` | 3700 mV |
| 状态 | `00` | 正常 |
| 序号 | `00 01` | 第 1 帧 |
| CRC | `4D 58` | CRC 值为 `0x584D` |

## 8. 数据清洗规则

| 字段 | 合法范围 |
|---|---|
| 温度 | -40.0 至 85.0 摄氏度 |
| 湿度 | 0.0% 至 100.0% |
| 气体浓度 | 0 至 5000 ppm |
| 电池电压 | 3000 至 4200 mV |
| 状态 | bit0-bit2 均为 0 时正常 |

任一字段异常时，JSON 中的 `valid` 为 `false`，并输出 `warnings` 数组。

## 9. JSON 示例

正常数据：

`{"device_id":16,"cmd":1,"temperature_c":25.3,"humidity_percent":60.8,"gas_ppm":300,"battery_mv":3700,"status":0,"sequence":1,"valid":true}`

MQTT Topic：

`sensor/16/data`

## 10. 流式解析策略

解析器内部保留尚未完成的字节，并按以下逻辑状态处理：

1. 查找第一帧头 `0xAA`。
2. 检查第二帧头 `0x55`。
3. 读取并校验 Payload 长度。
4. 等待完整命令字、设备 ID、Payload 和 CRC。
5. 执行 CRC16 校验。
6. 校验成功后输出 Frame。
7. 校验失败时丢弃错误起点并继续寻找下一帧。

因此解析器能够处理半包、粘包、CRC 错误、非法长度和前置噪声。
