# 模块实现说明

本文档说明网关各模块的职责、关键设计选择和当前工程边界。协议字段与测试步骤分别以 `protocol.md` 和 `test_cases.md` 为准。

## 1. 数据处理链路

网关的数据路径为：串口字节流 -> 环形缓冲区 -> 协议帧解析 -> CRC16 校验 -> 传感器字段解析与范围检查 -> JSON 序列化 -> MQTT/TCP 发布。

各阶段通过明确的数据结构和接口隔离。解析器只负责从字节流恢复协议帧，不直接访问网络；发布模块只接收已经序列化的消息，不依赖串口实现。

## 2. 串口与并发模型

每个串口设备由独立工作线程管理，使用 `termios` 配置原始模式、波特率和字符格式。线程负责阻塞读取、解析字节、处理读错误并执行重连。

串口断开时关闭失效文件描述符、清理未完成的解析状态并按配置间隔重新打开设备。进程收到 SIGINT 或 SIGTERM 后停止工作线程并关闭网络连接。

## 3. 协议解析

`FrameParser` 使用固定容量环形缓冲区保存跨越多次 `read()` 的数据。解析过程先定位 `0xAA 0x55`，再读取长度、命令字、设备 ID、Payload 和 CRC。

CRC 或长度校验失败时，解析器从错误起点之后继续搜索帧头，避免整段丢弃后续合法帧。诊断计数通过读取并清零接口交给应用层日志模块。

## 4. 传感器数据

命令 `0x01` 的 Payload 固定为 11 字节，包括温度、湿度、气体浓度、电池电压、状态位和序列号。多字节字段按大端顺序解析，温湿度使用 0.1 精度缩放。

数据清洗不会静默修改原始测量值。超出范围或状态位异常时，将 `valid` 标记为 `false` 并附加警告列表，随后仍可按配置上报以支持故障诊断。

## 5. MQTT 发布与缓存

MQTT 客户端基于 `libmosquitto`，使用长连接、独立网络循环和 QoS 1。发布成功以收到 PUBACK 为确认条件。

`ReliablePublisher` 在发布失败时将 Topic 和 JSON 写入 `MessageCache`。默认实现为 SQLite 队列，启用 WAL 和 `synchronous=FULL`；Broker 恢复后按入队顺序补传并在确认成功后删除。

该机制提供至少一次投递语义。发布确认与本地删除之间发生进程崩溃时可能重复发送，接收端应结合设备 ID 和序列号实现幂等处理。

## 6. TCP 发布

TCP 客户端基于 POSIX Socket，消息使用 JSON Lines 格式，以换行符界定每条 JSON。发送函数循环处理短写和 `EINTR`，连接失败不会阻止 MQTT 通道继续工作。

`PublisherGroup` 将同一条消息扇出到所有已启用通道，并分别记录每个通道的结果。

## 7. 配置与日志

`yaml-cpp` 负责加载串口、MQTT、TCP、缓存和日志配置。配置加载阶段执行类型、范围和必填项检查，错误配置会在启动阶段被拒绝。

日志模块提供 DEBUG、INFO、WARN、ERROR 四个等级，记录启动、连接状态、协议异常、数据校验、发布结果、缓存补传和退出过程。

## 8. 构建与部署

CMake 同时支持本机构建和 GNU AArch64 交叉编译。ARM64 脚本使用独立 sysroot，构建后检查 ELF 架构和动态依赖，并通过 QEMU 执行启动路径校验。

systemd 单元负责后台运行、故障拉起和受限权限部署。版本号由根目录 `VERSION` 统一提供，部署包附带 SHA-256 校验文件。

## 9. 当前工程边界

- MQTT 可靠缓存仅覆盖 MQTT 通道，TCP 通道当前不做持久化重放。
- 至少一次投递允许重复消息，服务端需要幂等处理。
- 运行原始时序数据在本地受控保存；仓库只记录已脱敏、可复现且纳入公开复现证据台账的结论与方法。
- Python、socat、CI 和 QEMU 用于无硬件回归，不能替代电气层和板级稳定性验证。
