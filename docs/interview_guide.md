# 面试讲解与追问准备

本文档用于把项目讲清楚，而不是背诵术语。回答时先讲问题和约束，再讲设计，最后给出代码与测试证据。

## 1. 一分钟项目介绍

这个项目是 STM32 多传感器系统的 Linux 边缘侧扩展。STM32 或 Python 模拟器通过串口发送二进制帧，Linux C++ 网关使用 termios 接收字节流，通过环形缓冲区和增量协议解析器处理半包、粘包与异常帧，完成 CRC16 校验、传感器数据清洗和 JSON 封装。数据可以同时发布到 MQTT Broker 和 TCP Server；MQTT 离线时消息进入 SQLite 持久化队列，连接恢复后按序补传。项目使用 CTest、Python 和 Shell 做单元、集成与端到端测试，并通过 GitHub Actions 验证 Ubuntu 构建和 AArch64 交叉编译。

## 2. 核心数据链路

```text
STM32 / Python 模拟器
    -> termios 串口接收
    -> ByteRingBuffer
    -> FrameParser
    -> CRC16-Modbus
    -> SensorData 解析与范围检查
    -> JSON
    -> PublisherGroup
       -> ReliablePublisher -> MQTT QoS 1
       -> TcpPublisher      -> TCP JSON Lines
```

讲解时指出：串口是无消息边界的字节流，TCP 同样是字节流，因此两侧都不能假设一次 `read()` 对应一条完整消息。

## 3. 为什么使用环形缓冲区

串口读取可能一次只得到半帧，也可能一次得到多帧。若每次都对线性 `vector` 头部执行删除，会产生移动开销，并让缓冲区边界处理变复杂。

当前实现使用固定容量 `ByteRingBuffer`：

- 写指针和读指针循环移动；
- 未完成帧保留到下一次输入；
- 解析成功后只消费对应字节；
- 容量固定，避免输入异常时无界增长；
- 对回绕、半包和粘包分别有测试。

可追问：容量满了怎么办？

回答要点：固定容量意味着必须有明确的溢出策略。当前协议帧很短，256 字节可容纳多帧；输入长期超过消费能力时不能静默覆盖，应丢弃不可恢复数据、记录告警并重新寻找帧头。生产版本还应增加溢出计数指标。

## 4. 协议解析如何处理半包和粘包

协议格式为：

```text
AA 55 | LENGTH | CMD | DEVICE_ID | PAYLOAD | CRC_LO CRC_HI
```

解析器按以下顺序增量工作：

1. 搜索 `0xAA 0x55` 帧头并跳过前置噪声。
2. 读取长度并检查是否超过协议上限。
3. 等待命令字、设备 ID、Payload 和 CRC 全部到齐。
4. 对长度字段到 Payload 末尾执行 CRC16-Modbus。
5. 校验成功后输出 `Frame`；失败后从可疑起点继续寻找下一帧。

半包场景下，解析器不清空尚未完成的数据；粘包场景下，输出一帧后继续解析缓冲区中的剩余字节。

可追问：为什么不能 CRC 失败后直接清空全部缓冲区？

回答要点：错误帧后面可能紧跟合法帧。全部清空会扩大单个坏字节的影响范围，因此应尽量从下一候选帧头恢复同步。

## 5. CRC16 校验的边界

项目使用 CRC16-Modbus，初值 `0xFFFF`、多项式反转形式 `0xA001`，结果按低字节在前传输。CRC 能发现传输中的常见比特错误，但不提供身份认证和防篡改能力；需要安全性时应由 TLS 或消息认证机制负责。

常见错误是 Python 模拟器与 C++ 端对 CRC 覆盖范围或高低字节顺序理解不一致。项目通过独立 CRC 测试向量和完整帧测试避免两端“同时写错但彼此通过”。

## 6. SensorData 与数据清洗

Payload 解码为温度、湿度、气体浓度、电池电压、状态字和序号。解析层负责字节序与缩放，清洗层负责范围检查，并将 `valid` 和 `warnings` 写入 JSON。

这里选择“保留异常数据并标记”而不是直接丢弃，原因是异常值可能来自真实故障。上层既能看到原始观测，也能根据 `valid` 决定是否用于业务计算。

可追问：协议长度合法但 Payload 业务长度不对怎么办？

回答要点：协议层只证明帧结构完整，命令字对应的业务层还要检查精确 Payload 长度；不满足时记录 `invalid payload length`，不进入发布链路。

## 7. MQTT 可靠性设计

`MqttClient` 基于 libmosquitto 维护长连接和网络循环，使用 QoS 1 发布并等待 PUBACK。`ReliablePublisher` 在发布失败时把 topic 和 payload 写入 `MessageCache`，Broker 恢复后从队首按序重放。

默认缓存后端是 SQLite：

- WAL 提高异常退出后的恢复能力；
- `synchronous=FULL` 优先保证持久性；
- 成功收到发布确认后才删除队首；
- 失败时保留未确认消息；
- 提供旧文件缓存到 SQLite 的迁移脚本。

可追问：能否保证消息绝不重复？

回答要点：不能。QoS 1 和“确认后删除”实现的是至少一次投递。发布已到 Broker 但确认尚未持久化时进程崩溃，恢复后可能重发。服务端可使用设备 ID 与 sequence 做幂等去重。

## 8. MQTT 与 TCP 双通道

业务层依赖统一 `Publisher` 接口，`PublisherGroup` 根据 YAML 配置组合 MQTT 和 TCP 通道。同一 JSON 依次扇出到已启用通道，某一路失败只影响该通道，不阻止其他通道继续发送。

TCP 客户端使用 POSIX Socket，并在每条 JSON 后追加换行符形成 JSON Lines。这样服务端可以按行恢复消息边界，避免把一次 `recv()` 错当成完整业务消息。

可追问：为什么 MQTT 有离线缓存而 TCP 没有？

回答要点：当前版本把可靠补传责任集中在 `ReliablePublisher` 的 MQTT 路径，TCP 是独立的尽力发布通道。若业务要求 TCP 也可靠，应将持久化队列下沉到通用 Publisher 装饰器，并为每个通道独立记录确认状态。

## 9. 多串口与线程模型

配置文件可以声明多个串口。主程序为每个串口创建独立工作线程，每个线程拥有自己的文件描述符、解析器和重连状态，避免不同设备的字节流互相污染。发布组件和日志组件是跨线程共享资源，因此接口需要保证并发调用安全。

可追问：为什么不使用 epoll 管理所有串口？

回答要点：当前设备数量有限，每串口一线程实现直接、便于隔离重连状态。若串口数量显著增长，可改为非阻塞 I/O 加 epoll，并把解析器状态按文件描述符保存；这属于规模扩大后的权衡，而不是当前版本的必需复杂度。

## 10. 配置、退出与部署

YAML 配置集中管理串口、MQTT、TCP、缓存、日志和重连参数，启动时执行类型与范围校验，非法配置直接拒绝运行，避免带着部分默认值进入不确定状态。

SIGINT/SIGTERM 处理器只设置退出标志，业务循环观察标志后停止读取、关闭串口和网络连接并刷新日志。信号处理器内不执行复杂 I/O，避免调用非异步信号安全函数。

systemd 单元负责后台运行、故障拉起和统一日志管理；安装脚本生成服务配置，部署文档说明启动、停止和卸载流程。

## 11. 自动化测试分层

- 单元测试：CRC、帧结构、环形缓冲区、SensorData、配置、日志和退出信号。
- 组件测试：MQTT 客户端、SQLite 缓存、ReliablePublisher、PublisherGroup 和 TCP 客户端。
- 端到端 smoke test：虚拟串口到 MQTT、多串口并发、TLS、缓存迁移、优雅退出和 TCP 服务端。
- 构建验证：Ubuntu 原生构建、AArch64 交叉编译、ELF 架构检查、QEMU 启动与部署包校验。

CTest 的 28 个测试运行很快，适合每次提交；依赖进程、端口和虚拟串口的 smoke test 由 Shell 脚本编排，在 CI 中分步骤运行，失败时更容易定位。

## 12. 真实调试案例

### 案例一：串口断开后进程退出

最初 `read()` 返回 I/O 错误后主循环直接结束。修复后把错误区分为可重试中断与连接丢失；连接丢失时关闭旧文件描述符，按配置间隔重新打开设备，并在恢复后继续使用新的解析器状态。

### 案例二：Broker 离线后的消息恢复

先停止 Mosquitto，确认发布失败并写入缓存；恢复 Broker 后，网关周期检查缓存并按顺序发布，全部收到确认后缓存行数归零。测试同时检查订阅端收到消息，避免只验证“本地删除成功”。

### 案例三：干净 ARM64 构建失败

增量构建曾掩盖 CMake `try_compile` 没有继承 sysroot 变量的问题。通过清空构建目录复现后，把 `AARCH64_SYSROOT` 加入 `CMAKE_TRY_COMPILE_PLATFORM_VARIABLES`，使嵌套配置过程继承交叉编译平台变量，并在 CI 中加入独立 ARM64 作业。

## 13. 不要这样回答

- 不说“CRC 保证数据安全”，应说它用于差错检测。
- 不说“QoS 1 保证不重不丢”，应说至少一次投递，可能重复。
- 不说“通过 QEMU 就等于真机验证”，应明确真机长稳仍待完成。
- 不把一次 `read()` 或 `recv()` 当作一条完整消息。
- 不声称项目已有吞吐量、P99 时延或 7x24 小时运行数据。
