# STM32 下位机接入验证

## 1. 文档范围

项目已完成 STM32 下位机通过物理串口接入 Linux 网关的功能验证。本文件分别记录早期课题组硬件验证（Historical Lab Validation）和公开复现验证（Reproducible Reference Validation）的边界、软件参数、步骤与结论。

早期课题组 STM32 硬件验证已经完成。原始实验记录未随仓库提交，因此 MCU 完整料号、板卡编号、USB 转串口芯片序列号、测试日期和责任人等字段保留在课题组设备台账中。本文不使用公开复现平台的参数替代历史设备记录。

## 2. 验证环境

| 项目 | 本项目记录 |
| --- | --- |
| 下位机 | STM32 多传感器终端，具体 MCU 料号待从设备台账补录 |
| Linux 端 | 运行 `edge_gateway` 的 Linux 主机或 ARM64 开发板 |
| 物理链路 | STM32 UART -> USB 转串口/板载串口 -> Linux TTY |
| 波特率 | 115200 bit/s |
| 数据格式 | 8 数据位、无校验、1 停止位、无硬件流控 |
| 串口模式 | `termios` raw mode |
| 数据类型 | 温度、湿度、气体浓度、电池电压、设备状态、序列号 |

FIT IoT-LAB OpenM3 兼容 STM32 节点已作为公开复现验证平台，用于验证 STM32 兼容 UART 数据源与网关协议兼容性。OpenM3 不等同于早期课题组历史验证设备，历史设备的完整型号仍以待补录的设备台账为准。

## 3. 协议帧

帧格式如下：

| 字段 | 长度 | 说明 |
| --- | ---: | --- |
| 帧头 | 2 B | 固定 `0xAA 0x55` |
| Payload 长度 | 1 B | 传感器上报命令固定为 `0x0B` |
| 命令字 | 1 B | `0x01` 表示传感器数据 |
| 设备 ID | 1 B | 示例为 `0x10` |
| Payload | 11 B | 传感器字段 |
| CRC16 | 2 B | CRC16-Modbus，低字节在前 |

已用于回归的完整示例帧：

`AA 55 0B 01 10 00 FD 02 60 01 2C 0E 74 00 00 01 4D 58`

该帧表示温度 25.3 摄氏度、湿度 60.8%、气体浓度 300 ppm、电池电压 3700 mV、状态 0、序列号 1。

## 4. 测试步骤

1. 按 115200 8N1 配置 STM32 UART 和 Linux 串口设备。
2. 使用 `stty` 或网关启动日志确认 TTY 设备可打开。
3. 启动 MQTT Broker、可选 TCP 测试服务端和 `edge_gateway`。
4. 由 STM32 周期发送合法传感器帧，确认 Linux 端收到原始字节并输出解析结果。
5. 对照下位机发送值与 JSON 字段，检查缩放、字节序、设备 ID 和序列号。
6. 连续发送多帧，并改变单次写入边界，验证半包和粘包处理。
7. 注入错误 CRC、非法长度和越界传感器值，检查过滤与诊断日志。
8. 断开并恢复串口连接，确认网关重试、重连并继续解析后续完整帧。

## 6. 异常现象与处理

- USB 转串口设备重新枚举后，原 TTY 节点可能短暂消失。网关关闭失效文件描述符并按配置间隔重新打开设备。
- 单次 `read()` 可能只返回部分协议帧，也可能包含多帧。解析器使用环形缓冲区保留未完成字节并连续提取完整帧。
- 线路干扰或下位机复位可能产生前置噪声、CRC 错误或非法长度。解析器记录分类计数并重新搜索帧头。
- 串口恢复时残留半帧不应与新连接数据拼接，重连路径会重置解析器状态。

## 7. 结论

STM32 下位机与 Linux 网关之间的物理串口链路、协议兼容性、异常帧处理和断线恢复已完成历史实验室验证。OpenM3 兼容 STM32 节点已完成公开复现验证。两类验证相互补充但不互相替代；Python 模拟器与 `socat` 虚拟串口继续用于日常回归，但不作为物理串口电气层验证的替代。
